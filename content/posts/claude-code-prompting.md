## 為什麼 Prompt 的品質很重要？

Claude Code 的輸出品質與你提供的 prompt 品質直接相關。一個模糊的指令可能讓 AI 做出你不想要的假設；一個清晰的指令則能讓 AI 精準理解你的意圖。

這篇文章整理了我在實際開發中測試過的技巧。

## 技巧一：提供足夠的背景資訊

**差的 prompt：**
```
幫我新增一個登入功能
```

**好的 prompt：**
```
這是一個 Next.js 14 App Router 專案，使用 TypeScript。
幫我在 app/auth 下新增登入頁面，需要：
- Email + 密碼登入表單
- 使用 app/globals.css 中的 CSS 變數 (--canvas, --vermillion 等)
- 與現有 Navbar 元件的視覺風格一致
- 暫時先用 mock 驗證，不需要實際後端
```

## 技巧二：指定輸出格式與限制

告訴 Claude Code 你**不想要**什麼，有時比說你想要什麼更有效：

```
重構 components/Navbar.tsx：
- 保持現有的功能不變
- 不要引入新的外部套件
- 不要改變元件的 props 介面
- 只調整樣式以符合 dark theme
```

## 技巧三：分步驟執行複雜任務

對於大型任務，拆分成步驟效果更好：

```
步驟一：先分析現有的 app/courses 頁面結構，告訴我你理解了什麼，不要做任何修改
```

確認理解正確後再繼續：

```
步驟二：根據你剛才的分析，將課程卡片改為可展開的詳細描述
```

## 技巧四：要求 Claude Code 解釋決策

```
在 lib/posts.ts 中實作快取機制，
並解釋你選擇這個方案的原因，以及有哪些替代方案
```

這樣你能學到更多，也能評估輸出的品質。

## 技巧五：利用現有程式碼作為範本

```
參考 components/Footer.tsx 的程式碼風格，
用相同的模式建立一個 Sidebar 元件
```

Claude Code 能分析你現有的程式碼風格並保持一致性。

## 常見錯誤與避免方法

| 錯誤 | 問題 | 改善方式 |
|------|------|---------|
| 太模糊 | "修一下這個 bug" | 提供完整錯誤訊息與預期行為 |
| 太寬泛 | "優化整個專案" | 指定具體的檔案或功能 |
| 沒有限制 | 不說明技術棧 | 明確說明語言、框架、版本 |
| 一次太多 | 同時要求10個功能 | 一次一個任務 |

## 我的模板庫

以下是我常用的幾個模板，可以直接複製使用：

**新功能開發**
```
專案：[框架] + [語言]
任務：[清楚描述要做什麼]
限制：[不能改什麼、不能引入什麼]
參考：[有哪些現有元件/風格可以參考]
```

**除錯**
```
錯誤訊息：[完整的 error log]
發生情境：[什麼操作會觸發這個錯誤]
預期行為：[應該要發生什麼]
已嘗試：[你試過哪些方法]
```

**程式碼審查**
```
請審查 [檔案路徑]，特別關注：
1. 潛在的效能問題
2. TypeScript 類型安全性
3. 與專案現有風格的一致性
```

## 結語

好的 prompt 是一種技能，需要不斷練習和調整。最重要的心態是：**把 Claude Code 當成一位聰明但需要清晰指示的新同事**，你給的資訊越完整，它的輸出就越符合你的需求。
